# 棱镜车智-专利论文数据穿透

## 使用时机

当用户要求进行汽车竞品配置分析、未来车型规划或配置导入时机判断，并且需要用 Patsnap 专利/文献/供应链证据解释建议背后的“为什么现在”时，使用本 Skill。

典型请求：

- "20-30 万 B 级 EREV SUV 哪些配置已经成为必备,哪些还有差异化机会"
- "对比理想 L6/L7、问界 M7/M8、零跑 C16、阿维塔 07、智界 R7,输出下一代 B-SUV 配置机会地图"
- "年轻家庭 25 万级 B-SUV,哪些配置该跟进,哪些适合作为专利壁垒型差异化"
- "电子外后视镜、地暖、二排腿托、座椅抽屉,哪些未来 6-18 个月值得布局"
- "把特斯拉、比亚迪、华为系作为头部标尺,看下一代智能座舱/舒适配置机会"

不要扩展成宽泛情报工作。如果用户询问公司战略、融资、组织、FTO 或精确 BOM/TCO，应将任务收窄回：用专利/文献/供应链证据支撑下一代车型配置决策。

## 核心受众

报告必须包含可见的“数据聚焦与服务对象”章节。说明本分析面向：

- 产品规划：定义下一代配置包和车型款型逻辑。
- 战略/竞品情报：对标市场头部并识别时机窗口。
- 研发预研：识别技术路线、专利、文献信号和设计约束。
- 采购/供应链：发现 OEM/Tier0/Tier1/Tier2 关联线索和寻源风险。
- 售前/管理层 Demo：快速传达决策结论和证据链。

## 数据新鲜度和预测规则

本 Skill 面向前瞻判断。始终使用执行环境中可获得的最新数据。

- 在 hero 区展示 `数据统计截止日期`、`Patsnap 数据截止日期` 和 `外部网页获取日期`。
- 优先使用尽可能接近报告日期采集或刷新的数据。
- 如果来源过旧，展示“前瞻性影响说明”卡片并降低信心。
- 必须披露专利公开滞后。不要把近期未见专利误判为没有活动。
- 对 1-2 年预测，综合市场配置搭载情况、Patsnap 专利/文献趋势和供应链成熟度。

## 核心工作流

1. **定义竞品池**：价格带、细分市场、动力类型、上市窗口、目标人群、车型清单和头部公司标尺。
2. **加入头部公司参考**：在细分市场允许时，纳入或明确讨论 Tesla、BYD、Huawei/Harmony Intelligent Mobility，以及相关中国新能源或海外头部企业。如果排除某一家，说明原因。
3. **解析产品域层级**：如果提示锁定了智能底盘、智能座舱、热管理、电池/电驱、座椅舒适、ADAS 等产品域，先映射产品域 → 一级技术 → 二级配置/零部件 → 分析对象。
4. **执行配置对标**：比较座椅、内饰、外饰、座舱、ADAS/舒适、热管理、电池和安全相关配置。保留用户指定车型池；头部公司仅作为标尺。
5. **构建 QFD / 质量屋**：将分级痛点转化为客户需求、工程特性、竞品满足度、Patsnap 证据强度、目标值和规划动作。
6. **执行 Matrix Analysis**：综合渗透率、QFD 权重、生命周期阶段、专利壁垒、供应链成熟度和头部公司标尺，归类为 must_follow / differentiation_bet / watch / avoid_or_design_around。
7. **应用 Patsnap 专利论文仪表盘 + 技术渗透墙**：对产品域地图中的每个一级技术建立专利/文献池，并展示市场现象 → 年度专利/文献趋势 → 关键申请人 → 共同申请人/发明人信号 → 核心专利亮点 → 问题-方案-效果 → 供应链可行性 → 1-2 年预测。
8. **应用供应链关联视角**：映射配置 → 功能模块 → 关键部件 → 核心技术 → OEM/Tier0/Tier1/Tier2 → 中国/海外区域 → 合作信号 → 证据来源。
9. **形成一句话决策**：创建 `headline_conclusion`，用一句简洁话说明未来机会、风险或行动。
10. **渲染 HTML**：使用 `assets/report_template.html` 和 `assets/vehicle_visuals/` 下的本地资产。图片生成或网页图片抓取只是可选增强，不是硬依赖。

## 必需理论标签

每个主要结论必须带至少一个理论标签。最终方法论展示应以 QFD / 质量屋 和 Matrix Analysis 为重点；其他方法作为辅助标签。

- `[QFD / House of Quality]`
- `[Matrix Analysis]`
- `[Benchmarking]`
- `[Kano]`
- `[S-Curve]`
- `[Patent Barrier]`
- `[Patsnap Penetration]`
- `[Supply Linkage]`

最终报告必须包含可见的“理论依据 / Methodology”章节，让非专业读者看到分析有方法支撑。

## 证据规则

- 市场事实优先使用官方或高可信来源。提出来源主张前先读取 `references/data_sources.md`。
- 对 Patsnap 驱动的主张，读取 `references/patsnap_data_lens.md`，并在主报告中展示完整渗透链，而不只是放入附录。
- 对供应商或供应链主张，读取 `references/supply_chain_lens.md`；供应商发现必须有来源支撑，或明确标注为“待验证供应商线索”。
- 不要用专利数量代替销量、用户需求或实际搭载率。
- “差异化下注”至少需要一个 Patsnap 强信号：专利趋势、文献前瞻、Tier1/Tier2 供应链证据，或具有明确设计意图的关键专利。
- “必须跟进”需要市场或用户证据：高渗透率、官方配置、明确 VOC 需求，或安全/法规依据。
- “规避设计/避免”必须说明理由是专利壁垒、用户排斥、法规不确定性、成本还是实施风险。
- 关键配置的专利证据必须可追溯到公开号、申请人、申请/公开日、可得法律状态、工具/来源和数据截止日期。
- OEM-供应商关系必须标记为：已验证合作、专利联合申请、产品/公告线索、待验证供应商线索。
- 如果数据源不可用，输出数据缺口卡并降低信心。绝不编造有来源支撑的数字。

## 头部公司标尺规则

车辆对标时，不要遗漏国内外头部企业：

- 默认头部公司标尺：Tesla、BYD、Huawei/Harmony Intelligent Mobility。
- 当产品域或市场范围相关时，加入 Volkswagen、Toyota、Hyundai、Mercedes-Benz 或其他海外头部企业。
- 头部公司是标尺，不替代用户指定的竞品池。
- 如果 Tesla、BYD 或 Huawei/Harmony Intelligent Mobility 不适用，需在报告中明确说明排除原因。

## 产品域优先规则

当用户点名或暗示某个产品域时，报告必须先展示产品域地图，再判断配置：

`产品域 → 一级技术 → 二级配置/零部件 → 竞品配置 → 专利/文献/供应链证据 → 规划动作`

典型产品域：智能座舱、智能底盘、ADAS/安全、热管理、电池/电驱、座椅与舒适、外饰与车身电子。

## 专利论文仪表盘和专利池规则

- 不要把原始专利清单作为主分析输出。
- 按公司、产品域和一级技术使用专利论文仪表盘：年度专利增长、文献/论文信号、申请人集中度、共同申请人网络、发明人信号、核心专利亮点、专利池规模和规划含义。
- 产品域地图中识别的每个一级技术都必须有对应技术渗透卡。如果报告分析六个一级技术，就渲染六张卡片；不要只展示部分示例。
- 每张技术卡必须包含产品域、一级技术、关联二级配置/零部件、专利池数量、检索窗口、检索词/同义词、文献/论文数量与信号、布局集群分布、代表性专利、申请人/供应商角色拆分、1-2 年预测、信心和数据缺口。
- 如果文献/论文检索无结果，明确显示“论文信号暂无，以专利为主”并继续。不要因文献数据不可用而生成失败。
- 将专利池视为该一级技术完整检索集合。主报告展示摘要指标和 3-6 件代表性专利；完整专利和文献清单放在详情页或同页附录。
- 推荐静态输出结构：主报告为 `index.html`，每个一级技术一个详情页 `patent_pools/{technology_slug}.html`。如果宿主 Agent 只能生成一个 HTML 文件，则降级为 `#pool-{technology_slug}` 等同页锚点，同时保留相同内容结构。
- 专利池详情页必须展示完整专利池、可用时的完整文献/论文池、布局集群、代表性专利解释、来源说明、数据缺口，以及可用的 Patsnap/来源详情页链接。
- 不要为每件单独专利生成一个 HTML 页面；这太慢且不必要。单件专利应链接到 Patsnap/来源详情 URL。
- 核心专利卡必须包含公开号、申请人、申请/公开日、可得法律状态、技术亮点、产品规划含义、可用详情 URL，以及 LLM 限制说明。
- 发明人信号可以提示技术连续性、团队聚集或重复研究方向，但不得单独用于推断雇佣关系、商业合作或供应商关系。

## HTML 生成和可移植规则

- 生成报告必须为可离线打开的静态 HTML，并对打包资产使用相对路径。
- 不依赖外部脚本、CDN 图表库或远程图片。
- 使用 `assets/report_template.html` 作为主要视觉骨架。`index.html` 和专利池详情页都复用其 CSS。
- 主报告使用摘要图表：sparkline、布局集群条、申请人角色卡和热力图。长表格和完整池放入详情页或同页附录。
- 模板必须随数据量伸缩：丰富数据集不应被截断；稀疏数据集仍应渲染数据缺口卡并降低信心。
- 确保响应式行为：桌面端可使用两列或卡片网格；移动端应退化为单列，长表格横向滚动，长文本自动换行。

## 交付级报告规则

当请求 HTML 输出时，默认生成交付级多页报告，而不是简陋的工作流验证页。

- Hero 右侧必须包含：主视觉、标题/状态标签、四张方法卡（QFD、Matrix、Patent、Lifecycle），以及三个或更多测试/报告摘要指标。
- 每个核心章节必须包含至少一种视觉表达：本地图片、静态 CSS/SVG 图表、关系卡、热力图、证据卡或紧凑仪表盘。不要让长章节只剩纯文字。
- 产品域技术地图必须渲染为表格，包含：产品域、一级技术、二级配置/零部件、相关车型/标尺、专利/文献池、供应商角色和规划动作。
- 配置渗透热力图必须包含数据缺口列，区分官方确认、公开页面确认、未验证、需授权配置库确认。
- 主 Patsnap 章节必须为每个一级技术包含 3-5 张代表性专利 PSE 卡。PSE 指技术问题、方案、预期效果、产品规划含义和风险/规避设计建议。
- 在专利论文渗透墙之后添加“AI 前瞻创新发现”模块。基于专利池、文献信号和目标场景推荐 3-5 个前瞻配置想法。每个想法必须说明相关技术域、发现依据、车型适配构想、1-2 年可行性、供应链成熟度、风险和建议动作。
- 专利池详情页必须包含本地视觉区块、检索范围、布局集群、带详情 URL 的代表性专利、完整专利清单、完整文献清单、申请人/供应商视图和数据缺口说明。
- 每行专利在可用时应链接到 Patsnap/来源详情。每行论文/文献在可用时应链接到来源 URL；否则显示“详情链接待补”。

## 图片稳定性和打包规则

最终报告图片在报告移动或压缩后必须仍可见。

- 最终交付前，将 `assets/vehicle_visuals/` 复制到报告输出目录。
- 主报告图片路径必须使用 `assets/vehicle_visuals/...`。
- 专利池详情页图片路径必须使用 `../assets/vehicle_visuals/...`。
- `data-fallback` 路径必须遵循与 `src` 相同的相对路径规则。
- 最终图片路径不要使用 `/Users/...`、`/tmp/...`、`../prism_auto_config_map/...` 或远程图片 URL。
- 写实资产和 SVG 兜底资产只是打包视觉素材，不是证据，不得描述为官方 OEM 图片。

## 真实测试输出规则

用于真实 Skill 测试和演示重跑时：

- 每次测试运行都创建新的输出目录。除非用户明确要求，否则不要覆盖旧报告。
- 优先输出 `index.html` + `patent_pools/*.html` + `assets/` + zip 包。
- 清晰命名交付测试包，例如 Skill 包用 `prism_auto_config_map_0529_delivery_upgraded.zip`，生成报告用 `{scenario}_report.zip`。
- 交付前验证 HTML 解析、本地多页链接、图片路径存在、zip 内容，以及解压后的离线可移植性。

## 技术生命周期策略

将每项关键配置或技术分类为：

- 引领技术：设定标杆、高端信号，通常渗透率较低但战略价值强。
- 上升技术：采用率上升、用户价值明确，适合作为卖点。
- 成熟技术：供应稳定、广泛适配，适合作为基线。
- 接近退出技术：可用于成本控制，但存在体验或品牌风险。
- 退出技术：不应出现在高端或下一代旗舰推荐中。

定价指导：

- 高端产品：必须包含引领技术作为标杆，上升技术作为卖点，成熟技术作为基础，不使用退出技术。
- 中端产品：用上升 + 成熟技术构建性价比竞争力。
- 低端产品：用成熟 + 接近退出技术控制成本，并明确披露体验和品牌风险。

## 输出结构

默认输出为使用 `assets/report_template.html` 的本地 HTML 报告。报告必须包含以下章节：

1. 带 `headline_conclusion` 的 Hero 决策摘要
2. 数据新鲜度和前瞻性提示
3. 数据聚焦与服务对象
4. 方法论重点：QFD / 质量屋 + Matrix Analysis，并摘要展示辅助标签
5. 产品域技术地图
6. 竞品池和头部公司标尺
7. 配置渗透热力图
8. 带痛点等级的 QFD / 质量屋
9. Matrix Analysis 机会板
10. Patsnap 专利论文仪表盘 + 技术渗透墙，包含每个一级技术的专利池
11. OEM-供应商-专利关系图，包括有证据支持的海外供应链线索
12. 技术生命周期 × 价格带策略矩阵
13. 下一代配置推荐 + 证据附录

在聊天中回复时，提供简洁摘要和生成的 HTML 路径。如果未请求或无法生成 HTML，仍按同样顺序组织 Markdown。

如果支持多页 HTML 生成，输出：

- `index.html`：主决策报告，包含每个一级技术的摘要卡。
- `patent_pools/{technology_slug}.html`：每个一级技术一页专利/文献池详情页。

如果不支持多页生成，则保留单个 `index.html`，并在同页锚点下渲染相同详情内容。

## 视觉资产规则

- 优先使用 `assets/vehicle_visuals/photo_real/` 中的本地写实资产。
- `assets/vehicle_visuals/` 下的本地 SVG 资产只作为低保真兜底。
- 选择视觉素材前读取 `assets/vehicle_visuals/metadata.json`。
- 按品牌、市场、动力类型或场景匹配。如果没有精确匹配，使用最接近的通用场景。
- 如果没有可用图片资产，渲染 CSS/图表占位并继续。
- 如图片生成或网页图片抓取可用，可作为增强手段。记录来源类型和使用说明；不得因为图片生成不可用而让 Skill 失败。
- 不要声称本地 AI 生成写实图或 SVG 视觉为官方 OEM 图片。
- 写实资产仅用于视觉增强，不得作为分析证据。

## 质量门

最终交付前验证：

- 报告在显著 hero 位置包含 `headline_conclusion`。
- 报告展示数据截止日期和任何前瞻性限制。
